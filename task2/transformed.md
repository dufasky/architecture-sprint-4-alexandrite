Для системы «Александрит» предлагается **комбинированный подход**, так как разные части системы требуют разных метрик:

- **USE (Utilization, Saturation, Errors)** – для инфраструктурных компонентов (базы данных, очереди, серверы).
- **RED (Rate, Errors, Duration)** – для API (Shop, CRM, MES).
- **Четыре золотых сигнала (Traffic, Errors, Latency, Saturation)** – для критичных бизнес-процессов (RabbitMQ, расчёт стоимости в MES).

### ****Метрики для мониторинга****

#### ****1\. RabbitMQ (USE + Четыре золотых сигнала)****

- **Number of dead-letter-exchange letters**
  - **Зачем**: Показывает количество неудачно обработанных сообщений.
  - **Ярлыки**: queue_name, error_type.
- **Number of messages in flight**
  - **Зачем**: Помогает выявить перегрузку очереди.
  - **Ярлыки**: queue_name.

#### ****2\. API (RED)****

- **Number of requests (RPS)** для Shop, CRM, MES API
  - **Зачем**: Оценка нагрузки на сервисы.
  - **Ярлыки**: endpoint, http_method.
- **Response time (latency)**
  - **Зачем**: Выявление медленных запросов.
  - **Ярлыки**: endpoint, http_method.
- **Number of HTTP 200/500**
  - **Зачем**: Контроль ошибок.
  - **Ярлыки**: endpoint, http_method.

#### ****3\. Базы данных (USE)****

- **Memory Utilisation** для Shop DB и MES DB
  - **Зачем**: Предотвращение нехватки памяти.
  - **Ярлыки**: db_name.
- **Number of connections**
  - **Зачем**: Выявление перегрузки БД.
  - **Ярлыки**: db_name, user_type.
- **Size of DB instance**
  - **Зачем**: Контроль роста данных.

#### ****4\. Серверы (USE)****

- **CPU % и Memory Utilisation** для Shop, CRM, MES API
  - **Зачем**: Мониторинг ресурсов серверов.
  - **Ярлыки**: service_name, host.

#### ****5\. Дополнительные метрики****

- **Size of S3 storage**
  - **Зачем**: Контроль за хранением 3D-файлов.
- **Kb transferred (received/sent)**
  - **Зачем**: Анализ сетевой нагрузки.
  - **Ярлыки**: api_name.

**План действий (драфт ТЗ)**

**1\. Развертывание системы мониторинга**

- **Создать инстанс Prometheus + Grafana** для сбора и визуализации метрик.
- **Настроить экспортеры** для:
  - RabbitMQ (через rabbitmq-exporter).
  - Баз данных (PostgreSQL-exporter).
  - API (метрики из Spring Boot, C#, Vue/React).
- **Добавить Alertmanager** для уведомлений о критичных инцидентах.

**2\. Интеграция с CI/CD**

- **Добавить мониторинг в пайплайн**:
  - Проверка метрик после деплоя в dev/release.
  - Автоматические алерты при аномалиях (например, рост 500-х ошибок).

**3\. Оптимизация RabbitMQ**

- **Настроить кластеризацию** для отказоустойчивости.
- **Добавить retry-логику** для обработки сбоев.

**4\. Мониторинг бизнес-метрик**

- **Внедрить дашборды для операторов**:
  - Количество новых заказов в реальном времени.
  - Среднее время расчёта стоимости в MES.

**5\. Резервное копирование и логирование**

- **Настроить ELK-стек** (Elasticsearch, Logstash, Kibana) для логов.
- **Автоматизировать бэкапы БД** (например, через WAL-G для PostgreSQL).

**Через полгода система мониторинга должна:**

Отслеживать **критичные ошибки в реальном времени** (RabbitMQ, API, БД).  
Позволять **оперативно выявлять узкие места** (например, медленные запросы в MES).  
**Интегрироваться с CI/CD** для предотвращения деплоя проблемных версий.  
**Давать бизнесу прозрачность** (статусы заказов, нагрузка на производство).

**Первые три шага:**

1. Развернуть Prometheus + Grafana.
2. Настроить мониторинг RabbitMQ и API.
3. Добавить алерты для HTTP 500 и высокой загрузки CPU.

Эти изменения быстро снизят количество инцидентов и улучшат стабильность системы.
