## **Системы для сбора логов**

Компоненты, требующие логирования:

- **Internet Shop** (Vue + Spring Boot)
- **CRM** (Vue + TS)
- **MES API** (C#)
- **RabbitMQ** (очереди сообщений)
- **Shop DB** (PostgreSQL)
- **MES DB** (PostgreSQL)

## **2\. Список логов (уровень INFO)**

| **Система** | **Логируемые события** | **Пример данных** |
| --- | --- | --- |
| **Internet Shop** | Создание заказа (SUBMITTED), загрузка 3D-файла (FILE_UPLOADED) | {time: "2025-05-21T10:00", user_id: 123, order_id: 456, status: "SUBMITTED"} |
| **CRM** | Изменение статуса заказа (например, MANUFACTURING_APPROVED) | {order_id: 456, old_status: "SUBMITTED", new_status: "APPROVED", actor: "seller-789"} |
| **MES API** | Начало/завершение расчёта стоимости (PRICE_CALCULATED) | {order_id: 456, calculation_time_sec: 180, polygons_count: 5000} |
| **RabbitMQ** | Доставка/потеря сообщения (dead-letter) | {message_id: "msg-xyz", queue: "orders", status: "DELIVERED"} |
| **Shop DB** | Долгие запросы (>1 сек) | {query: "SELECT \* FROM orders", duration_ms: 1200} |

### ****Другие уровни логирования****

- **DEBUG**: Для разработки (например, тело запросов между сервисами).
- **WARN**: Подозрительные события (повторные попытки отправки в RabbitMQ).
- **ERROR**: Критические сбои (500-е ошибки API, недоступность БД).

## **3\. Мотивация**

**Зачем нужно логирование?**

1. **Технические выгоды**:
    - Быстрое выявление причин сбоев (например, почему заказ «завис» в CRM).
    - Анализ производительности (долгие запросы к БД).
2. **Бизнес-метрики**:
    - Время выполнения заказа (от SUBMITTED до SHIPPED).
    - Количество отменённых заказов из-за ошибок.
    - Нагрузка на операторов (логи статусов в MES).

## **4\. Приоритеты внедрения**

Из-за ограниченных ресурсов начните с:

1. **CRM + RabbitMQ** (ключевое место потери заказов).
2. **MES API** (долгий расчёт стоимости).
3. **Internet Shop** (точка входа клиентов).

Трейсинг сначала внедряется для **MES API и RabbitMQ**, так как они критичны для бизнеса.

## **5\. Предлагаемое решение**

### ****Технологии****

- **ELK-стек** (Elasticsearch + Logstash + Kibana) для хранения и анализа.
- **Fluentd** или **Filebeat** для сбора логов с серверов.
- **Grafana** для дашбордов (интеграция с Elasticsearch).

### ****Новые компоненты****

1. **Logstash** (агрегация и фильтрация логов).
2. **Elasticsearch** (хранилище).
3. **Kibana** (визуализация).
4. **Fluentd** (агенты на серверах).

### ****Безопасность****

- **Доступ**: Только для DevOps и поддержки (ролевая модель в Kibana).
- **Чувствительные данные**: Маскировка PII (номера карт, emails) в Logstash.
- **Шифрование**: TLS для передачи логов.

### ****Политика хранения****

- **Горячие данные**: 7 дней (быстрый доступ).
- **Холодные данные**: 30 дней (архив в S3).
- **Размер**: ~50 ГБ/месяц (оценка для 10K заказов/день).

## **6\. Анализ логов и алертинг**

- **Алерты**:
  - Резкий рост ошибок 500 в API (возможна атака).
  - Dead-letter в RabbitMQ >10 за 5 мин.
- **Аномалии**:
  - Необычная активность (например, 10K заказов за секунду).
  - Долгие (>5 мин) расчёты стоимости в MES.
